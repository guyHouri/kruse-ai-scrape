# Jack Kruse forum scraper — first iteration: pinned threads

## Context

This repo is a partial export of the "Food For AI" project, originally built to scrape Hebrew Australia/NZ/Fiji travel sites for a NotebookLM notebook. The user (Guy, account `guy-houri` on the forum) wants to repurpose it to scrape **forum.jackkruse.com** — a XenForo v2.x health/biology discussion forum — into a NotebookLM-ready markdown bundle.

A previous attempt by a friend hacked the existing `website_to_md/` pipeline by:
- Adding a `kruse` entry to `websites.json` pointing at a single thread (`maxDepth: 1`)
- Hardcoding logged-in XenForo cookies into [`website_to_md/code/crawler.js:36`](website_to_md/code/crawler.js:36) and [`website_to_md/code/extractor.js:39`](website_to_md/code/extractor.js:39)

That attempt produces broken output:
- `links/kruse.txt` has 161 URLs, mostly forum chrome (`/whats-new/`, `/account/`, `/help/`, `/conversations/`, `/online/`, `/members/`, reply/quote/bookmark/react endpoints, `/posts/N/*` action URLs, page-2 pagination, and per-post anchor URLs `/threads/X/post-N` that all redirect back to the same thread page)
- `processed_mds/website-kruse-2026-05-11.md` is 670KB of which the first article is the **forum homepage menu chrome**, and the subsequent "articles" are massively duplicated (every `/post-N` URL re-renders the same thread page)
- `CONTENT_SELECTORS` picks `<article>` first; XenForo wraps each post in `<article class="message">`, so the extractor grabs only the first post and drops the rest of the thread

The architecture is calibrated for "article-portal" sites (one URL = one article). Forums are structurally different (one logical document = one thread spanning N paginated pages, each page contains M post blocks, each block carries author + date + permalink). Forcing it through `website_to_md/` would muddy that part's design.

**Goal of this plan:**
1. Build a clean, forum-aware sibling project-part: `forum_to_md/`
2. First iteration target: **all pinned (sticky) threads across every subforum** — these are Jack's canonical content
3. Output: ONE big markdown file with strong, machine-detectable thread boundaries so the user can split into per-thread files later if NotebookLM source count becomes a problem
4. Each post inside a thread = its own subsection with author, ISO date, and `#post-N` permalink so NotebookLM can cite individual posts
5. Lay groundwork for a v2 discovery mode: "every thread Jack Kruse has posted in" (the user's wildest-dream scope) — but don't implement it in this iteration

---

## Recommended approach: new sibling project-part `forum_to_md/`

**Why a new folder, not modifying `website_to_md/`:**
- The two parts have fundamentally different "atomic unit" semantics: website = page, forum = thread (with N pages, M posts)
- Output filename convention diverges (`website-<slug>-<date>.md` vs `forum-jackkruse-<date>.md`); the existing `cleanup.js` regex is slug-specific
- Mixing forum logic into `website_to_md/` would force conditional branches throughout (`is this a forum URL? then…`) — ugly and error-prone
- Matches the existing pattern: `whatsapp_to_md/`, `facebook_group_to_md/`, `whatsapp_auto_exporter/` are each their own project-part
- The existing 6 inactive Australia entries in `websites.json` stay parked exactly as they are, in case the friend wants them back

## Stage 1 — Discover (`code/discover.js`)

1. Fetch `FORUM_BASE_URL` (the forum index page)
2. Parse `.node--forum` items → list of subforum URLs (`/forums/<slug>.<id>/`) with their human-readable names. The forum index already has these because the friend's previous run captured the markdown of it.
3. For each subforum (concurrency-limited):
   - Fetch page 1 of the subforum
   - Parse thread list rows: each thread is a `.structItem.structItem--thread` row
   - **Sticky detection**: a sticky thread row carries either `class="... is-sticky"` on the wrapper OR `<i class="structItem-statusSticky" title="Sticky">` inside `.structItem-statuses`. Use both checks defensively across XenForo template variations.
   - Harvest `{ url, title, subforum }` for each sticky
4. Dedupe by thread id (extracted from URL: `/threads/<slug>.<id>/` → id is the trailing numeric)
5. Write `forum_to_md/threads.json`:
   ```json
   [
     {
       "id": 32507,
       "url": "https://forum.jackkruse.com/threads/the-human-gps-system-defines-how-systems-operate-in-humans.32507/",
       "title": "The Human GPS System Defines How Systems Operate in Humans",
       "subforum": "Optimal Reset",
       "source": "pinned",
       "discovered_at": "2026-05-11T..."
     },
     ...
   ]
   ```
6. Log per-subforum counts + grand total. Expected: ~30–100 pinned threads across all subforums.

**Note on the friend's existing run:** the previous `website_to_md` crawl harvested the forum index into the bloated `website-kruse-2026-05-11.md`. We don't reuse that — we re-fetch fresh and parse the live HTML directly, which is cheap (~1 request per subforum).

## Stage 2 — Extract (`code/extract.js` + `code/xenforo.js`)

For each entry in `threads.json` (concurrency-limited):

1. Fetch thread page 1 → parse `.pageNav` to find total page count (or absent → single page)
2. Fetch pages 2..N sequentially (within the per-thread fetch — across threads, multiple threads run in parallel up to CONCURRENCY)
3. For each page, parse `article.message` blocks. For each post extract:
   - **postId**: `data-content` attribute (`post-NNNNN`) or `id` attribute on the `article`
   - **author**: `.message-name a` text (or `[itemprop="name"]`)
   - **date**: `time.u-dt` `datetime` attribute (ISO 8601) — fall back to `<time>` `data-time` or the visible text
   - **permalink**: `.message-attribution-main a[href]` (resolved against thread URL) — XenForo emits `/goto/post?id=NNNNN` and `/threads/<slug>.<id>/post-NNNNN` variants; prefer the latter
   - **body HTML**: `.message-body .bbWrapper` (or `.message-userContent .bbWrapper`)
4. Run body HTML through Turndown (reuse the same config as `website_to_md/code/extract.js:12-25`: ATX headings, fenced code, `-` bullets, strip `<img>`)
5. Inside the body, preserve `<blockquote class="bbCodeBlock bbCodeBlock--quote">` as markdown blockquotes — Turndown does this by default, but verify nested quotes are sensible (XenForo wraps quotes with a "Click to expand" toggle; the inner content is what we want)
6. Drop posts whose body markdown is < `MIN_POST_BODY_CHARS` (typical: "+1", emoji-only reactions)
7. Dedupe by postId across pages (defensive — pagination can overlap by one row on some XenForo configs)
8. Sort posts by their natural page-order (XenForo serves posts oldest-first within a page; concatenate in page order)

Then build the master markdown file (see "Output format" below) and write to `processed_mds/forum-jackkruse-<YYYY-MM-DD>.md`.

**Cleanup before write**: `code/cleanup.js` deletes any prior `forum-jackkruse-*.md` so only the latest dated file remains (mirrors `website_to_md/code/cleanup.js:4-21` pattern).

## Output format

```markdown
---
slug: jackkruse-forum
site: forum.jackkruse.com
discovery_mode: pinned
scraped_at: 2026-05-11T14:32:00.000Z
total_threads: 87
total_posts: 1342
---

# Jack Kruse Optimal Health Forum — Pinned Threads

Aggregated pinned threads from forum.jackkruse.com. Each thread is bounded
by a `═══...` separator line. Each post inside a thread is a `### <Author> — <Date>`
subsection with a clickable permalink back to the original post.

══════════════════════════════════════════════════════════════════════
# Thread: The Human GPS System Defines How Systems Operate in Humans
**Thread URL:** <https://forum.jackkruse.com/threads/the-human-gps-system-defines-how-systems-operate-in-humans.32507/>
**Subforum:** Optimal Reset
**Posts:** 28
══════════════════════════════════════════════════════════════════════

### Jack Kruse — 2024-03-15T14:32:00Z
**Source:** <https://forum.jackkruse.com/threads/the-human-gps-system-defines-how-systems-operate-in-humans.32507/post-370317>

<post body markdown — including quoted blockquotes, lists, links, code>

### riksa — 2024-03-16T09:11:00Z
**Source:** <https://forum.jackkruse.com/threads/the-human-gps-system-defines-how-systems-operate-in-humans.32507/post-370318>

<post body markdown>

### Jack Kruse — 2024-03-16T12:04:00Z
**Source:** <https://forum.jackkruse.com/threads/the-human-gps-system-defines-how-systems-operate-in-humans.32507/post-370319>

> riksa said:
> <quoted text>

<Jack's reply markdown>

══════════════════════════════════════════════════════════════════════
# Thread: <Next pinned thread title>
**Thread URL:** ...
...
```

**Why this format works for the user's "one big file, splittable later" requirement:**
- The `═══...` line at column 0 is a unique grep target — splitting by it later is one line of awk/JS
- Each thread block starts with a `# Thread:` H1 (in addition to the separator) — also splittable
- Each post is a `### <Author> — <Date>` H3 — NotebookLM will surface these as chunk headers, giving citations like "in *The Human GPS System...* by Jack Kruse on 2024-03-15"
- Per-post `**Source:**` URL means a NotebookLM retrieval can link directly to the exact post

   - Expect: `threads.json` populated with 30–100 entries, each with a real `subforum` field

   - Confirm no `/whats-new/` / `/account/` / `/members/` URLs leaked in
3. **Extract only**: `npm start -- --extract-only` (or `npm run extract`)
   - Expect: one file `processed_mds/forum-jackkruse-<today>.md`
   - File size sanity: rough estimate 50 threads × ~10 posts × ~1KB = ~500KB. If <50KB something's wrong; if >5MB something's bleeding chrome.
4. **Output spot-checks** (use Read with `offset`/`limit`, per repo memory rule):
   - Top of file: YAML header has correct `total_threads` and `total_posts`
   - Search for `═══` to count thread separators → matches `total_threads`
   - Pick one thread known to span multiple pages → confirm post count matches what the live forum shows
   - Pick one quoted reply → confirm the quote renders as a markdown blockquote
   - Confirm no `Menu` / `Install the app` / `out of date browser` chrome bleeds into post bodies
5. **Re-upload to NotebookLM**: replace the previous `website-kruse-2026-05-11.md` source with the new `forum-jackkruse-<date>.md`
6. **Full run**: `npm start` (discover + extract chained)
